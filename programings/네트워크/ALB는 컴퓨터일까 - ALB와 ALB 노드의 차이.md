
AWS에서 로드밸런서를 설정하다가 의문이 생겼다. ALB는 트래픽을 받아서 EC2나 ECS로 전달하는 역할인데, 트래픽을 감지하려면 결국 컴퓨터가 있어야 하는 거 아닌가? 그렇다면 ALB는 컴퓨터인 건가?

---

## ALB는 컴퓨터인가?

결론부터 말하면, ALB는 **논리적 설정**이고 실제 트래픽을 처리하는 건 **ALB 노드**라는 컴퓨터다.

- ALB = AWS 콘솔에서 생성하는 설정 묶음 (리스너, Target Group, 헬스체크 등)
- ALB 노드 = 위 설정을 실제로 실행하는 AWS 내부 EC2

즉, 내가 ALB를 만들면 AWS가 알아서 내부 EC2 여러 대를 띄워서 트래픽을 처리한다. Nginx 같은 리버스 프록시가 돌아가는 컴퓨터라고 생각하면 된다.

## ALB vs ALB 노드

|  | ALB | ALB 노드 |
|---|---|---|
| 정체 | 설정/서비스 이름 | AWS가 관리하는 EC2 |
| 개수 | 1개 | 여러 대 (AZ당 최소 1대) |
| IP | 없음 (DNS 이름만 존재) | 있음 (유동 IP) |
| 관리 | 내가 콘솔에서 설정 | AWS가 자동 생성/삭제 |

```
ALB (논리적 설정)
├── 리스너 규칙 (80 → HTTPS, 443 → Target Group)
├── Target Group (내 ECS 컨테이너들)
├── 헬스체크 설정 (/health, 30초 간격)
└── DNS 이름: my-alb-1234.ap-northeast-2.elb.amazonaws.com

이 설정을 실제로 실행하는 것이 ↓

ALB 노드들 (AWS가 관리하는 EC2)
├── 노드-A (AZ-a, IP: 52.1.2.3)
├── 노드-B (AZ-c, IP: 52.4.5.6)
└── (트래픽 증가 시 노드-C 자동 추가)
```

## ALB는 왜 고정 IP가 아닌가?

처음에는 ALB도 컴퓨터니까 고정 IP를 가질 수 있지 않나? 라고 생각했다.

알아보니 ALB는 단일 컴퓨터가 아니라 **여러 대의 노드**로 운영된다. 트래픽이 늘면 노드를 추가하고, 줄면 제거한다. 노드가 바뀌니 IP도 바뀌는 것이다.

- 노드 수가 변한다 — 트래픽에 따라 2대 → 5대 → 3대
- 장애 시 노드 교체 — 문제 있는 노드를 새 노드로 교체하면 IP가 달라진다
- AWS가 의도적으로 이렇게 설계 — 고정 IP에 묶이면 스케일링/장애 복구가 불가능

비유하자면 이렇다.
- EC2 = 내가 직접 고용한 직원 1명 (고정 자리, 고정 번호)
- ALB = 콜센터 대표번호 (내부 상담원 수가 수시로 바뀌지만, 대표번호는 고정)

그래서 ALB는 고정 IP 대신 **고정 DNS 이름**(`*.elb.amazonaws.com`)을 제공한다.

> 참고: 고정 IP가 반드시 필요하면 **NLB Network Load Balancer**를 사용한다. NLB는 AZ당 1개의 고정 IP(또는 Elastic IP)를 할당받을 수 있다.

## 트래픽은 어떻게 분산되는가?

ALB 노드가 여러 대인데, 사용자의 요청은 어떤 노드로 가는 걸까?

DNS가 ALB의 DNS 이름을 조회하면 **노드들의 IP 목록**을 반환한다.

```
$ nslookup my-alb.elb.amazonaws.com

응답:
52.1.2.3    ← 노드-A
52.4.5.6    ← 노드-B
52.7.8.9    ← 노드-C
```

DNS는 매번 IP 순서를 섞어서 반환하고, 브라우저는 보통 첫 번째 IP를 선택한다.

```
사용자-A → DNS 조회 → [52.1.2.3, 52.4.5.6] → 노드-A
사용자-B → DNS 조회 → [52.4.5.6, 52.1.2.3] → 노드-B
```

이게 **DNS 라운드 로빈**이라는 1차 분산 방식이다.

## 전체 요청 흐름

```
사용자가 api.example.com 으로 요청
         ↓
Route 53 (DNS) → ALB 노드의 IP로 해석
         ↓
ALB 노드 (AWS 내부 EC2) → 요청을 받아서 헬스체크 통과한 ECS에 전달
         ↓
ECS 컨테이너 (내 Spring Boot 앱) → 처리 후 응답
         ↓
ALB 노드 → 사용자에게 응답 반환
```

ALB 노드에는 내 코드가 전혀 없다. 오직 **"요청 받고 → 살아있는 ECS에 전달"**만 한다.

## ALB와 함께 동작하는 서비스들

ALB 혼자 모든 걸 하는 건 아니다. 각자 역할이 나뉘어 있다.

| 서비스 | 역할 |
|---|---|
| ALB | 헬스체크 + 살아있는 EC2/ECS에 트래픽 분산 |
| CloudWatch | EC2/ECS 지표(CPU, 메모리 등) 모니터링 + 알람 발생 |
| Auto Scaling Group | CloudWatch 알람 기반으로 EC2/ECS 수 자동 조절 |

```
CloudWatch (모니터링) → "CPU 80% 초과!"
         ↓
Auto Scaling Group → ECS 컨테이너 추가 생성
         ↓
Target Group에 자동 등록
         ↓
ALB 헬스체크 통과 → 트래픽 분산 시작
```

특정 컴퓨터 1대가 관리하는 게 아니라, 여러 매니지드 서비스가 협력해서 동작하는 구조다.

## 도메인 연결

ALB를 생성하면 AWS가 자동으로 DNS 이름을 부여한다.

```
my-alb-1234567890.ap-northeast-2.elb.amazonaws.com
```

이 주소로 바로 접속할 수 있다. 도메인은 이 긴 주소를 `api.example.com`처럼 짧게 바꾸는 것이다.

Route 53에서 도메인을 연결할 때는 **Alias 레코드**를 사용한다. A 레코드(IP 직접 지정)가 아니라 Alias를 쓰는 이유는, ALB의 IP가 유동적이기 때문이다. Alias는 AWS 내부에서 ALB DNS를 직접 참조하므로 노드 IP가 바뀌어도 자동 반영된다.

## 정리

| 개념 | 설명 |
|---|---|
| ALB | AWS 콘솔에서 만드는 논리적 설정 |
| ALB 노드 | 설정을 실행하는 실제 컴퓨터 (AWS 관리 EC2) |
| 유동 IP | 노드 수가 변하므로 IP도 변한다 |
| DNS 이름 | ALB의 고정 식별자 (`*.elb.amazonaws.com`) |
| DNS 라운드 로빈 | IP 목록 순서를 섞어서 노드 간 트래픽 분산 |
| Alias 레코드 | Route 53에서 ALB에 도메인을 연결하는 방식 |
| NLB | 고정 IP가 필요할 때 사용하는 로드밸런서 |

핵심은 ALB를 "컴퓨터 1대"로 생각하면 안 된다는 것이다. ALB는 설정이고, 실제 일하는 건 AWS가 자동으로 관리하는 여러 대의 ALB 노드다.
