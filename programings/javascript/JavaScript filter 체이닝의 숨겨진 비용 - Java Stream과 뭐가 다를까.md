
프론트엔드 작업을 하다가 배열에 `.filter()`를 두 번 걸어야 하는 상황이 생겼다.

조건이 두 개라 자연스럽게 `.filter().filter()`로 체이닝했는데, 문득 의문이 들었다.

Java에서는 Stream으로 `.filter()`를 아무리 많이 걸어도 중간 컬렉션이 생기지 않았는데, JavaScript도 그럴까?

## 결론부터 말하면, 다르다

JavaScript의 `.filter()`는 호출할 때마다 **즉시 전체를 순회하고 새 배열을 만든다.**

```javascript
const result = arr
    .filter(x => x > 10)      // 순회 1번 + 새 배열 생성
    .filter(x => x % 2 === 0); // 순회 1번 + 새 배열 생성
```

배열이 10만 개면, 최악의 경우 중간 배열도 10만 개짜리가 된다. 순회도 두 번, 배열 생성도 두 번이다.

반면 Java의 Stream은 이렇게 동작하지 않는다.

```java
list.stream()
    .filter(x -> x > 10)
    .filter(x -> x % 2 == 0)
    .collect(Collectors.toList());
```

Stream은 **지연 평가 Lazy Evaluation** 방식이다. 중간 연산인 `.filter()`는 호출 시점에 아무것도 하지 않고, 최종 연산인 `.collect()`가 실행될 때 파이프라인을 합성해서 **한 번만** 순회한다. 중간 컬렉션도 없다.

## 왜 이런 차이가 생길까??

핵심은 **즉시 평가 Eager Evaluation** vs **지연 평가 Lazy Evaluation**의 차이다.

| 구분 | JavaScript Array | Java Stream |
|------|-----------------|-------------|
| 평가 방식 | 즉시 평가 (Eager) | 지연 평가 (Lazy) |
| `.filter()` 호출 시 | 바로 순회 + 새 배열 반환 | 아무것도 안 함 (파이프라인에 추가만) |
| 중간 컬렉션 | 매번 생성 | 없음 |
| 순회 횟수 | 체이닝 수만큼 | 최종 연산 시 1번 |

JavaScript의 Array 메서드들은 설계 자체가 "호출하면 바로 결과를 돌려준다"는 철학이다. 그래서 `.filter()`, `.map()`, `.reduce()` 모두 즉시 실행된다.

Java의 Stream은 반대로 "최종 연산 전까지는 아무것도 하지 않는다"는 철학이다. 중간 연산은 그냥 레시피를 쌓아놓는 것일 뿐이다.

## 그러면 JavaScript에서는 어떻게 해야 할까??

가장 간단한 건 조건을 하나로 합치는 것이다.

```javascript
// 순회 1번, 배열 생성 1번
const result = arr.filter(x => x > 10 && x % 2 === 0);
```

조건이 복잡해서 분리하고 싶다면 `reduce`로 한 번에 처리할 수도 있다.

```javascript
const result = arr.reduce((acc, x) => {
    if (x > 10 && x % 2 === 0) acc.push(x);
    return acc;
}, []);
```

## 근데 프론트엔드니까 무조건 최적화해야 하는 거 아닌가??

이것도 생각해봤는데, 꼭 그렇지는 않다.

프론트엔드에서 진짜 비용이 큰 건 **DOM 조작과 렌더링**이다. 배열을 두 번 도는 건 수 마이크로초 차이지만, 불필요한 리렌더링 한 번이 수십 밀리초를 잡아먹을 수 있다.

React나 Vue에서 성능 최적화의 핵심은 `useMemo`, `computed` 같은 메모이제이션이나 가상 DOM diffing 최소화 쪽이지, `.filter()` 체이닝을 줄이는 게 아닌 경우가 대부분이다.

프론트엔드 성능 최적화의 우선순위를 정리하면 이렇다:

1. 불필요한 리렌더링 방지
2. 네트워크 요청 최적화
3. 대량 데이터의 연산 최적화

`.filter()` 체이닝 문제는 3번에 해당하는데, 배열이 수백~수천 개 수준이면 사실상 차이가 없다.

다만 그리드에 수만 건의 데이터를 뿌리는 경우라면 얘기가 달라진다. `.filter().map().sort()` 이런 체이닝을 걸면 대량 데이터에서는 체감이 될 수 있고, 이런 상황에서는 합치는 게 맞다.

## 정리

| 구분 | 내용 |
|------|------|
| JavaScript `.filter()` | 즉시 평가 - 호출할 때마다 새 배열 생성 |
| Java Stream `.filter()` | 지연 평가 - 최종 연산 시 한 번만 순회 |
| 해결법 | 조건을 `&&`로 합치거나 `reduce`로 한 번에 처리 |
| 프론트에서 중요한 건 | `.filter()` 최적화보다 리렌더링 방지가 우선 |
| 대량 데이터라면 | 체이닝 합치는 게 실질적으로 의미 있음 |

결국 "프론트니까 무조건 아껴야 한다"보다는, **병목이 어디인지 파악하고 거기에 집중하는 게** 더 정확한 방향이다.
